Hidden 是什么？

Hidden 是我利用业余时间做的一款 App，它可以帮助用户隐藏想要隐藏的 App。

不同于 iOS 桌面编辑中的「从主屏幕移除」，Hidden 可以真正的让 App 处于一种被卸载拟态，比如同时隐藏掉 App 相关的桌面小组件，隐藏掉 App 相关的快捷指令，桌面搜索甚至 App 资源库也无法搜索到被隐藏起来的 App。

Emmm…这种感觉就好像我之前描述的，貌似处于一种被卸载状态，但不是真的被卸载，可以随时还原让 App 重新出现。

你可以把 Hidden 当作一个隐私工具，在他人借用手机时隐藏掉私密 App；你可以把 Hidden 当作一个效率工具，在工作或者集中注意力时隐藏分散你注意力的 App；你可以把 Hidden 当作一个家长管理工具，必要时将那些使用时长过多对孩子健康造成影响的 App 隐藏起来。

总之，基于隐藏应用的核心功能可以派生出好多有趣的用法，因此 Hidden 未来的迭代方向也有很多。

核心概念

  • 授权，Hidden 的主要能力是隐藏和锁定 App，这项能力来自于 ScreenTime API，所以在没有得到授权时 Hidden 几乎没有任何用，这也是 Hidden 在用户未授权 ScreenTime 时向用户申请权限的原因。

  • 隐私，ScreenTime API 的设计非常注重隐私安全，Hidden 不会知道你安装了哪些 Apps，以及什么 Apps 是你想要隐藏起来的。其实 Hidden 的使用并不需要网络授权，目前请求网络权限也仅仅是为了同步 AppStore 的信息。

  • 限制，Hidden 只会对没有 ScreenTime API 授权的 App 生效，这意味着 Hidden 不可以隐藏自己，所以不是很有趣。这也意味着如果你想取消 Hidden 的控制，只需要收回 Hidden 拿到的 ScreenTime 授权或者干脆卸载 Hidden，那些被 Hidden 控制起来的 Apps 就会一切如初。

设计理念

  • 应用分组，管理单个应用比较麻烦，所以 Hidden 加入了分组的概念，以卡片形式展示。卡片信息没有展示内部有哪些 Apps，因为 Hidden 真的不知道组内有什么 Apps。市面上有些类似的产品可以展示用户选择了什么 Apps（猜测是通过 ScreenTime API 乱码形式的 AppToken 做匹配实现的），这违反了 Apple 对于屏幕时间授权的使用建议，由于存在被吊销授权的风险，我暂时不打算跟进这个 Feature。

  • 身份认证，看到 Hidden 内容的人应该是设备的主人，我比较懒只做了生物识别认证，也就是 Face ID 或 Touch ID。因为 iOS16 兼容的设备（ScreenTime 本设备授权最低系统版本）似乎没有不支持生物识别硬件能力的设备了，我实在没有动力做一套 Hidden 自建的密码验证机制（想要越过 Hidden 限制的人如果可以通过生物识别，也就意味着 Ta 可以通过系统设置收回 Hidden 的 ScreenTime 权限，所以自建密码验证毫无意义）。

  • 限制卸载，如果 Hidden 不再那么小众，那么可能拿到你设备的人即使不进行身份认证依然可以通过卸载 Hidden 的方式看到那些你不愿意让 TA 看到的 Apps，所以 Hidden 做了这个功能；相应的还有「限制安装应用」和「限制应用内购买」功能，笔者认为这三个功能对于监护人更好的约束被监护人这个场景很有用，可以防止孩子安装新的手游或私下购买游戏道具/打赏女主播。

无可奈何

  • 图标乱序，被 Hidden 隐藏的 Apps 再次出现时会按照某种规则重新出现到设备桌面，就好像它们被 Hidden 隐藏的时真的被卸载然后出现时又重新从 AppStore 安装一样。暂时无解，不过可以通过在第二屏预留相应空位或者参照 Hidde 内首次切换隐藏模式时的提示绕过去。当然，你也可以试试通过 Hidden 的锁定模式来禁用 App，锁定不会乱序所以适用于自律等非必要隐藏的场景。

  • 界面简陋，呃… 因为目前还没找到合作的设计师，所以整个 App 的交互和界面设计都是我自己做的，确实有点偏性冷淡的素。其实仔细体会的话你也许可以发现所有的交互路径都是尽可能短的，目的自然是为了更高效，界面元素也基本都是简单的图形更多的是对代码逻辑的直接映射。再多说两句，其实 AppIcon 的 H 不只是 Hidden 的首字母，也是一个栅栏的形象，意味着约束和控制；主题色是绿色和蓝色的混色，表示健康/距离感与可靠。

未来计划

正如文章开头说的那样，基于隐藏应用的核心功能可以派生出好多有趣的用法，因此 Hidden 未来的迭代方向也有很多。老实说，我目前对于 Hidden 的规划更倾向于效率类工具，所以后续大概率会往这个方向做迭代。

如果你有好的想法建议或者问题反馈也非常欢迎联系我（推荐使用 Email，前段时间登陆微博发现一条两年前的私信，汗），有价值的建议和想法会得到实现，一经采纳还会视情况回馈小礼物（暂时没想好什么礼物就是了）。

What is Hidden?

Hidden is an app that I made in my spare time that helps users hide apps they want to hide.

Unlike “Remove from Home Screen” in iOS desktop editing, Hidden can truly put apps into an uninstalled state, such as hiding related desktop widgets, shortcuts for App-related instructions, desktop searches or even App resource libraries cannot search for hidden Apps.

Emmm… This feeling seems like what I described before - it seems to be in an uninstalled state but not really uninstalled. It can be restored at any time so that the App reappears.

You can think of Hidden as a privacy tool to hide private apps when others borrow your phone; you can think of it as an efficiency tool to hide distracting apps while working or focusing attention; you can think of it as a parental management tool to hide Apps that have been used too much and may affect children’s health if necessary.

In short, based on the core function of hiding applications, many interesting uses can be derived from them. Therefore, there are many future directions for iteration on Hidden.

Core Concepts

  • Authorization: The main ability of Hidden is to hide and lock Apps. This ability comes from the ScreenTime API. Therefore, without authorization, Hidden is almost useless. This is also why Hidden applies for permission to users who have not authorized ScreenTime.

  • Privacy: The design of the ScreenTime API pays great attention to privacy and security. Hidden does not know which Apps you have installed or what Apps you want to hide. In fact, using Hidden does not require network authorization at present, and requesting network permissions only synchronizes AppStore information.

  • Limitation: Hidden only works on Apps that do not have ScreenTime API authorization. This means that Hidden cannot hide itself, so it is not very interesting. It also means that if you want to cancel the control of Hidden, just revoke the ScreenTime authorization obtained by Hidden or simply uninstall it, and those Apps controlled by hidden will return to their original state.

Design Philosophy

  • Application grouping: Managing individual applications can be cumbersome, so Hidden introduces the concept of grouping in card form display. The card information does not show which Apps are inside because Hidden really doesn’t know what’s inside each group of apps. Some similar products on the market can display what apps users choose (guessing matching through AppToken in garbled form through ScreenTime API), which violates Apple’s usage recommendations for screen time authorizations and may result in revocation of authorizations risk; therefore I temporarily don’t plan to follow up this feature.

  • Identity authentication: People who see content in Hidde should be device owners. I am lazy and only did biometric authentication - Face ID or Touch ID - because iOS16-compatible devices (the lowest system version for local device authorization) seem There are no devices without biometric hardware capabilities anymore; I really don’t have any motivation to make a password verification mechanism built into Hiddne (if someone wants to bypass hidden restrictions If they can pass biometrics, it means they can revoke Hiddne’s ScreenTime permission through system settings, so self-built password verification is meaningless).

  • Uninstall limitation: If Hidden is no longer so niche, people who get your device may still be able to see the Apps you don’t want them to see by uninstalling Hidden even if they do not authenticate. Therefore, Hidden has this function; corresponding functions also include “Limit installation of applications” and “Limit in-app purchases”, which I think are very useful for guardianship scenarios where parents need to restrict their children from installing new mobile games or buying game props/rewarding anchors.

Helplessness

  • Icon disorder: When hidden apps reappear, they will be rearranged on the device desktop according to some rules, as if they were really uninstalled when hidden and then reinstalled from the App Store when they appear again. There is currently no solution for this problem, but you can bypass it by reserving corresponding space on the second screen or referring to the prompt when switching to hide mode for the first time in Hidde. Of course, you can also try disabling apps through Hidden’s lock mode. Locking won’t cause disorder and is suitable for non-essential hiding scenarios such as self-discipline.

  • Simple interface: Um… because I haven’t found a cooperating designer yet, so I did all of Hidden’s interaction and interface design myself. It does have a bit of bias towards minimalism. If you carefully experience it, you may find that all interaction paths are as short as possible for higher efficiency. The interface elements are mostly simple graphics that directly map to code logic. To say more, actually “H” in AppIcon not only stands for “Hidden”, but also represents a fence image which means constraint and control; The theme color is a mixture of green and blue representing health/distance sense and reliability.

Future plans

As mentioned at the beginning of this article, many interesting uses can be derived from Hidden’s core function of hiding applications; therefore there are many directions for future iterations of Hidden. Honestly speaking, my current plan for Hidden leans more towards efficient tools; therefore subsequent iterations will probably move in this direction.
